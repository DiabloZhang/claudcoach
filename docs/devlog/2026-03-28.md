## 确立的工作哲学

**能不依靠用户就不依靠用户**

起因：我没有线上数据库的访问权，无法查到 2026-03-14 异常游泳的活动 ID。
错误做法：让用户手动查 ID，再手动调用排除接口。
正确做法：做一个 `/backfill` 接口，自动扫全量数据、自动排除、自动补字段；同时在同步流程里加入自动检测，以后新活动永不遗漏。

**判断标准**：
- 让用户"去找某个数据"→ 先想能不能写接口自动完成
- 让用户"手动执行某步骤"→ 先想能不能集成进现有流程
- 只有真正需要用户决策时才打断用户

这个哲学已记入 `CLAUDE.md`，下次对话自动加载。

## 遇到的问题

- PostgreSQL `Integer` 只支持到 ~21亿，Strava activity ID 已超过，需用 `BigInteger`
- PostgreSQL 不支持 SQLite 的 `PRAGMA table_info`，迁移逻辑需分别处理
- Railway PORT 变量会被平台自动覆盖，不能手动设置 PORT=3000（前端实际跑在 8080）
